================================================================================
一、项目概述
================================================================================

1. 目标硬件：ESP32-C3 Zero 开发板（4MB Flash，RISC-V 单核）
2. 开发框架：Arduino 框架，底层基于 ESP-IDF 原生 FreeRTOS 实时操作系统
3. 工程配置：通过 platformio.ini 统一管理，使用 SPIFFS 文件系统，支持双分区 OTA
4. 构建工具：使用根目录 build.py 脚本进行编译、上传、监控
5. 版本管理：Git，开发版本从 v0.0.1 开始

================================================================================
二、分层架构设计（标准五层 + 公共层）
================================================================================

从下到上共 6 层，上层只能调用下层，禁止跨层调用、禁止反向调用。

┌─────────────────────────────────────────────────────┐
│  App 应用层                                         │  业务逻辑、场景组合
├─────────────────────────────────────────────────────┤
│  Service 系统服务层                                 │  WiFi、MQTT、NTP、Web、OTA、存储
├─────────────────────────────────────────────────────┤
│  Driver 外设驱动层                                  │  LED、门锁、按键、传感器、RFID
├─────────────────────────────────────────────────────┤
│  BSP 板级支持层                                     │  GPIO、UART、PWM、I2C、SPI
├─────────────────────────────────────────────────────┤
│  Common 公共基础层                                  │  配置、日志、错误码、工具函数
├─────────────────────────────────────────────────────┤
│  Arduino / ESP-IDF 框架层                           │  系统底层（框架自带，不改动）
└─────────────────────────────────────────────────────┘

补充：RTOS 任务（rtos/ 目录）不属于某一层，而是横向贯穿各层的调度单元。


【各层职责】

1. Common 公共层（最底层，所有层都可调用）
   - 全局配置定义（引脚、波特率、任务参数、功能开关）
   - 统一错误码、日志宏、通用类型定义
   - 通用工具函数（CRC、字符串转换、字节操作）
   - 不包含任何业务逻辑

2. BSP 板级支持层
   - 芯片片内外设的二次封装（GPIO、UART、I2C、SPI、PWM）
   - 屏蔽芯片 SDK 的细节差异，向上层提供统一接口
   - 换芯片/开发板时，只需修改此层
   - 不允许包含业务逻辑

3. Driver 外设驱动层
   - 具体外部器件的驱动实现（LED、按键、门锁、传感器、RFID等）
   - 调用 BSP 层总线接口操作具体器件
   - 每个器件独立封装，可插拔
   - 器件之间尽量独立，互不依赖

4. Service 系统服务层
   - 系统级服务（WiFi、MQTT、NTP、Web、OTA、存储等）
   - 封装网络协议、文件系统、参数管理
   - 提供初始化 + 状态查询 + 数据收发接口
   - 服务内部可维护状态机，但不做业务决策

5. App 应用层（最上层）
   - 产品业务逻辑的实现
   - 状态机管理、权限判断、场景组合
   - 协调各 driver 和 service 完成产品功能
   - 不允许直接操作硬件寄存器
   - 不允许直接调用 SDK 底层 API


【分层调用约束】

✅ 允许：上层调用下层（App → Service → Driver → BSP → Common）
✅ 允许：同层模块之间调用（需注意解耦）
✅ 允许：任何层调用 Common 层
❌ 禁止：下层调用上层（反向依赖）
❌ 禁止：跨层调用（如 App 直接调用 BSP）
❌ 禁止：业务代码中直接操作硬件寄存器

四、RTOS 任务设计规范
================================================================================

1. 任务创建
   - 每个任务一个 .cpp 文件，.h 文件仅导出任务创建函数
   - 任务函数本身设为 static（不对外暴露）
   - 创建函数命名：task_<功能>_create()，返回 app_err_t
   - 所有任务在 main.cpp 中统一创建（或通过 app_main 统一调用）

2. 任务间通信
   - 任务之间只能通过 RTOS 通信原语交互
   - 禁止直接调用另一个任务的内部函数
   - 禁止用全局变量大量传递数据

   | 场景                | 推荐方式            |
   |---------------------|---------------------|
   | 数据传递（带内容）  | 消息队列 Queue      |
   | 事件通知（标志位）  | 事件组 EventGroup   |
   | 资源互斥（保护共享）| 互斥锁 Mutex        |
   | 简单同步（轻量）    | 信号量 Semaphore    |

3. 任务优先级（从高到低参考）
   - 硬件实时任务（如按键扫描）：较高
   - 通信任务（如 UART/MQTT）：中等
   - 业务逻辑任务：中等偏低
   - 状态指示任务（如 LED）：较低
   - 优先级数值统一在 rtos/task_config.h 中定义

4. 任务栈
   - 栈大小统一在 rtos/task_config.h 中配置
   - 以 configMINIMAL_STACK_SIZE 的倍数表示
   - 合理估算，避免溢出或浪费


================================================================================
五、命名规范
================================================================================

1. 文件命名
   - 目录名：全小写 + 下划线  例：bsp/、app_task/
   - 源文件：层级_功能.cpp    例：bsp_uart.cpp、srv_mqtt.cpp
   - 头文件：层级_功能.h      例：drv_led.h

2. 函数命名
   - 格式：层级_模块_动作()
   - 示例：
     - bsp_uart1_init()
     - drv_door_open()
     - srv_wifi_connect()
     - app_door_control_handle()

3. 宏定义
   - 全大写 + 下划线
   - 示例：UART1_BAUDRATE、MAX_DOOR_OPEN_TIME

4. 枚举与类型
   - 类型定义：名称_t 后缀  例：app_err_t、msg_type_t
   - 枚举值：全大写 + 下划线  例：MSG_TYPE_UART_DATA

5. 变量
   - 全局变量：g_ 前缀（尽量少用）  例：g_system_state
   - 静态变量：s_ 前缀              例：s_uart_rx_buf
   - 局部变量：小驼峰或下划线（项目内统一）

6. 错误码
   - 统一类型：app_err_t
   - 命名：APP_ 前缀
   - 示例：APP_OK、APP_FAIL、APP_ERR_TIMEOUT、APP_ERR_INVALID_PARAM


================================================================================
六、编码规范
================================================================================

1. main.cpp 规则
   - 只做硬件初始化和任务创建
   - 禁止在 main.cpp 中写业务逻辑
   - setup() 中初始化 + 创建任务，loop() 留空或做最小工作

2. 头文件规则
   - 所有头文件必须有 #pragma once 或 include guard
   - 头文件只放声明，不放实现（模板除外）
   - 尽量前向声明，减少头文件嵌套

3. 注释规范
   - 文件头注释：包含文件名、功能、作者、日期、版本
   - 函数注释：说明功能、参数、返回值
   - 复杂逻辑处加行内注释
   - 注释简洁明了，不写废话

4. 错误处理
   - 函数返回 app_err_t 表示执行结果
   - 重要操作必须检查返回值
   - 错误发生时记录日志（LOG_E）

5. 代码风格
   - 缩进：4 空格（或 2 空格，项目内统一）
   - 大括号：独占一行或同行，项目内统一
   - 每行长度：建议不超过 100 字符
   - 函数长度：建议不超过 50 行，过长则拆分

七、配置管理规范
================================================================================

1. 所有可配置参数集中在 common/config.h 中
   - 引脚号、波特率、地址等硬件参数

   - 功能开关宏（#define FEATURE_XXX_ENABLE 1）

3. 编译时配置

   - 不需要的功能可以编译时裁剪，减小固件体积

十一、LED 状态显示系统（已实现）
================================================================================

1. 硬件配置
   - LED 类型：WS2812 RGB LED（1颗）
   - 数据引脚：GPIO10
   - 驱动库：FastLED
   - 默认亮度：100（0~255）

2. 三个核心状态（优先级从高到低）

   | 状态          | 颜色  | 灯效     | 优先级 | 说明                           |
   |---------------|-------|----------|--------|--------------------------------|
   | ERROR         | 红色  | 快闪5Hz  | 3（最高）| 系统错误、连接失败             |
   | CONNECTING    | 蓝色  | 慢闪1Hz  | 2      | WiFi/MQTT/蓝牙 连接中          |
   | NORMAL        | 白色  | 呼吸灯   | 1（最低）| 正常运行、空闲                 |

3. 状态堆叠机制
   - 使用 set_state() 压入状态，clear_state() 弹出状态
   - 系统始终显示当前最高优先级的活跃状态
   - 多个模块可独立设置/清除自己的状态，互不覆盖丢失
   - 示例：正常运行时发起连接 → 显示蓝色慢闪；连接中发生错误 → 显示红色快闪；
           错误恢复 → 自动回到蓝色慢闪；连接成功 → 自动回到白色呼吸

4. 两层对外接口

   第一层：预设状态（业务语义，推荐使用）
     srv_led_set_state(LED_STATE_CONNECTING);   // 设置连接中状态
     srv_led_clear_state(LED_STATE_CONNECTING); // 清除连接中状态

   第二层：自定义灯效（灵活扩展）
     srv_led_solid(color);                      // 常亮
     srv_led_blink(color, freq_hz, duty);       // 闪烁
     srv_led_breathe(color, period_ms);         // 呼吸
     srv_led_off();                             // 灭灯

5. 软件分层（严格遵守五层架构）

   src/driver/drv_ws2812.h/.cpp   Driver层：FastLED封装 + 灯效引擎
   src/service/srv_led.h/.cpp     Service层：状态管理 + 优先级仲裁
   src/rtos/task_led.cpp          RTOS任务：周期刷新输出
   src/common/config.h            配置：引脚、数量、任务参数

6. 使用约定
   - 业务模块只调用 service 层接口，禁止直接操作 driver 层
   - 新增预设状态：在 led_state_t 加枚举，在 srv_led.cpp 状态表加一行配置
   - 新增灯效模式：在 drv_ws2812 加模式和算法，在 srv_led 加对外接口
   - 服务自动注册：SERVICE_REGISTER("led", srv_led_init, 10)
   - 上电默认状态：CONNECTING（蓝色慢闪），符合开机即连接的流程

7. 驱动层支持的灯效模式
   - LED_MODE_OFF      灭灯
   - LED_MODE_SOLID    常亮（参数：颜色）
   - LED_MODE_BLINK    闪烁（参数：颜色、周期、占空比）
   - LED_MODE_BREATHE  呼吸（参数：颜色、周期）

十二、WiFi 连接系统（已实现）
================================================================================

1. 总体设计
   - WiFi 作为 Service 层服务，通过 SERVICE_REGISTER 自动初始化
   - AP+STA 双模式同时开启，AP 始终可用，STA 按需连接
   - 状态机管理：IDLE / CONNECTING / CONNECTED / RETRY_WAIT / SCANNING
   - 指数退避重连：失败后等待时间翻倍，上限 30 秒
   - 与 LED 状态灯联动：连接中=蓝色慢闪，连接成功=回到白色呼吸
   - NVS 持久化存储：最多保存 3 条 WiFi 配置，超过覆盖最旧的（FIFO）

2. WiFi 状态机

   上电 → AP 启动 + 尝试连接已保存的 WiFi
     → 有保存配置 → CONNECTING（蓝色慢闪）
       → 连接成功 → CONNECTED（白色呼吸）
       → 连接失败 → RETRY_WAIT（保持蓝色慢闪 + 指数退避）
         → 等待时间到 → CONNECTING（循环）
     → 无保存配置 → IDLE（AP 模式等待配置）

3. AP 模式
   - SSID 格式：OPNC-XXXX（XXXX 为芯片 MAC 后 4 位十六进制）
   - 开放热点，无密码
   - 信道：6
   - 最大连接数：4
   - AP 模式始终开启，不受 STA 连接状态影响

4. 软件分层（严格遵守五层架构）

   src/service/srv_wifi.h/.cpp      Service层：状态机 + 事件回调 + NVS存储 + 扫描
   src/service/srv_webconfig.h/.cpp Service层：HTTP服务器 + DNS劫持 + 配置页
   src/rtos/task_wifi.cpp           RTOS任务：周期处理WiFi逻辑 + Web配置页请求
   src/common/config.h              配置：AP参数、NVS键名、超时、重连参数

6. Web配置页（Captive Portal）
   - 手机连上AP后自动弹出配置页（DNS劫持实现）
   - 访问地址：http://192.168.4.1/ 或任意域名
   - 功能：
     - 刷新按钮：扫描周围WiFi并列表显示
     - 点击WiFi：弹出密码输入框，输入后连接
     - 当前状态：实时显示连接状态和IP
   - 简洁工业风暗色主题，移动端适配
   - API 接口：
     - GET /api/scan      触发扫描并返回结果（JSON数组）
     - GET /api/connect   连接指定WiFi（ssid+pwd参数）
     - GET /api/sta       查询当前STA状态（JSON）

10. 使用约定
    - 业务模块只调用 service 层接口，禁止直接操作 WiFi.h
    - 配置参数统一在 config.h 中定义
    - WiFi 任务只做轻量处理（超时检测、重连触发、Web请求处理）
    - 实际连接逻辑在事件回调中异步处理
    - LED 状态联动在 srv_wifi 内部处理，外部模块不需要手动管 LED
    - NVS 存储键名格式：namespace=wifi，s0/p0 ~ s2/p2 + count
    - 已保存WiFi超过3条时，最旧的（索引0）被覆盖，新的放到末尾
