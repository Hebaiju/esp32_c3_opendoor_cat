                        ESP32 开门器项目 - 开发规则文档

创建日期：2026-06-27
适用项目：esp32-opendoor（ESP32-C3 开门器）

================================================================================
一、项目概述
================================================================================

1. 目标硬件：ESP32-C3 Zero 开发板（4MB Flash，RISC-V 单核）
2. 开发框架：Arduino 框架，底层基于 ESP-IDF 原生 FreeRTOS 实时操作系统
3. 工程配置：通过 platformio.ini 统一管理，使用 SPIFFS 文件系统，支持双分区 OTA
4. 构建工具：使用根目录 build.py 脚本进行编译、上传、监控
5. 版本管理：Git，开发版本从 v0.0.1 开始

================================================================================
二、分层架构设计（标准五层 + 公共层）
================================================================================

从下到上共 6 层，上层只能调用下层，禁止跨层调用、禁止反向调用。

┌─────────────────────────────────────────────────────┐
│  App 应用层                                         │  业务逻辑、场景组合
├─────────────────────────────────────────────────────┤
│  Service 系统服务层                                 │  WiFi、MQTT、NTP、Web、OTA、存储
├─────────────────────────────────────────────────────┤
│  Driver 外设驱动层                                  │  LED、门锁、按键、传感器、RFID
├─────────────────────────────────────────────────────┤
│  BSP 板级支持层                                     │  GPIO、UART、PWM、I2C、SPI
├─────────────────────────────────────────────────────┤
│  Common 公共基础层                                  │  配置、日志、错误码、工具函数
├─────────────────────────────────────────────────────┤
│  Arduino / ESP-IDF 框架层                           │  系统底层（框架自带，不改动）
└─────────────────────────────────────────────────────┘

补充：RTOS 任务（rtos/ 目录）不属于某一层，而是横向贯穿各层的调度单元。


【各层职责】

1. Common 公共层（最底层，所有层都可调用）
   - 全局配置定义（引脚、波特率、任务参数、功能开关）
   - 统一错误码、日志宏、通用类型定义
   - 通用工具函数（CRC、字符串转换、字节操作）
   - 不包含任何业务逻辑

2. BSP 板级支持层
   - 芯片片内外设的二次封装（GPIO、UART、I2C、SPI、PWM）
   - 屏蔽芯片 SDK 的细节差异，向上层提供统一接口
   - 换芯片/开发板时，只需修改此层
   - 不允许包含业务逻辑

3. Driver 外设驱动层
   - 具体外部器件的驱动实现（LED、按键、门锁、传感器、RFID等）
   - 调用 BSP 层总线接口操作具体器件
   - 每个器件独立封装，可插拔
   - 器件之间尽量独立，互不依赖

4. Service 系统服务层
   - 系统级服务（WiFi、MQTT、NTP、Web、OTA、存储等）
   - 封装网络协议、文件系统、参数管理
   - 提供初始化 + 状态查询 + 数据收发接口
   - 服务内部可维护状态机，但不做业务决策

5. App 应用层（最上层）
   - 产品业务逻辑的实现
   - 状态机管理、权限判断、场景组合
   - 协调各 driver 和 service 完成产品功能
   - 不允许直接操作硬件寄存器
   - 不允许直接调用 SDK 底层 API


【分层调用约束】

✅ 允许：上层调用下层（App → Service → Driver → BSP → Common）
✅ 允许：同层模块之间调用（需注意解耦）
✅ 允许：任何层调用 Common 层
❌ 禁止：下层调用上层（反向依赖）
❌ 禁止：跨层调用（如 App 直接调用 BSP）
❌ 禁止：业务代码中直接操作硬件寄存器

四、RTOS 任务设计规范
================================================================================

1. 任务创建
   - 每个任务一个 .cpp 文件，.h 文件仅导出任务创建函数
   - 任务函数本身设为 static（不对外暴露）
   - 创建函数命名：task_<功能>_create()，返回 app_err_t
   - 所有任务在 main.cpp 中统一创建（或通过 app_main 统一调用）

2. 任务间通信
   - 任务之间只能通过 RTOS 通信原语交互
   - 禁止直接调用另一个任务的内部函数
   - 禁止用全局变量大量传递数据

   | 场景                | 推荐方式            |
   |---------------------|---------------------|
   | 数据传递（带内容）  | 消息队列 Queue      |
   | 事件通知（标志位）  | 事件组 EventGroup   |
   | 资源互斥（保护共享）| 互斥锁 Mutex        |
   | 简单同步（轻量）    | 信号量 Semaphore    |

3. 任务优先级（从高到低参考）
   - 硬件实时任务（如按键扫描）：较高
   - 通信任务（如 UART/MQTT）：中等
   - 业务逻辑任务：中等偏低
   - 状态指示任务（如 LED）：较低
   - 优先级数值统一在 rtos/task_config.h 中定义

4. 任务栈
   - 栈大小统一在 rtos/task_config.h 中配置
   - 以 configMINIMAL_STACK_SIZE 的倍数表示
   - 合理估算，避免溢出或浪费


================================================================================
五、命名规范
================================================================================

1. 文件命名
   - 目录名：全小写 + 下划线  例：bsp/、app_task/
   - 源文件：层级_功能.cpp    例：bsp_uart.cpp、srv_mqtt.cpp
   - 头文件：层级_功能.h      例：drv_led.h

2. 函数命名
   - 格式：层级_模块_动作()
   - 示例：
     - bsp_uart1_init()
     - drv_door_open()
     - srv_wifi_connect()
     - app_door_control_handle()

3. 宏定义
   - 全大写 + 下划线
   - 示例：UART1_BAUDRATE、MAX_DOOR_OPEN_TIME

4. 枚举与类型
   - 类型定义：名称_t 后缀  例：app_err_t、msg_type_t
   - 枚举值：全大写 + 下划线  例：MSG_TYPE_UART_DATA

5. 变量
   - 全局变量：g_ 前缀（尽量少用）  例：g_system_state
   - 静态变量：s_ 前缀              例：s_uart_rx_buf
   - 局部变量：小驼峰或下划线（项目内统一）

6. 错误码
   - 统一类型：app_err_t
   - 命名：APP_ 前缀
   - 示例：APP_OK、APP_FAIL、APP_ERR_TIMEOUT、APP_ERR_INVALID_PARAM


================================================================================
六、编码规范
================================================================================

1. main.cpp 规则
   - 只做硬件初始化和任务创建
   - 禁止在 main.cpp 中写业务逻辑
   - setup() 中初始化 + 创建任务，loop() 留空或做最小工作

2. 头文件规则
   - 所有头文件必须有 #pragma once 或 include guard
   - 头文件只放声明，不放实现（模板除外）
   - 尽量前向声明，减少头文件嵌套

3. 注释规范
   - 文件头注释：包含文件名、功能、作者、日期、版本
   - 函数注释：说明功能、参数、返回值
   - 复杂逻辑处加行内注释
   - 注释简洁明了，不写废话

4. 错误处理
   - 函数返回 app_err_t 表示执行结果
   - 重要操作必须检查返回值
   - 错误发生时记录日志（LOG_E）

5. 代码风格
   - 缩进：4 空格（或 2 空格，项目内统一）
   - 大括号：独占一行或同行，项目内统一
   - 每行长度：建议不超过 100 字符
   - 函数长度：建议不超过 50 行，过长则拆分


================================================================================
七、配置管理规范
================================================================================

1. 所有可配置参数集中在 common/config.h 中
   - 引脚号、波特率、地址等硬件参数
   - 任务栈大小、优先级、队列长度
   - 功能开关宏（#define FEATURE_XXX_ENABLE 1）
   - 超时时间、阈值等业务参数

2. 禁止在代码中硬编码魔法数字
   - 用宏定义或 const 常量代替
   - 统一在 config.h 中管理

3. 编译时配置
   - 通过 #define 开关可选功能
   - 不需要的功能可以编译时裁剪，减小固件体积


================================================================================
八、构建与版本约定
================================================================================

1. 构建工具
   - 统一使用根目录 build.py 脚本
   - 命令：
     python build.py build      # 仅编译
     python build.py upload     # 编译 + 上传
     python build.py monitor    # 串口监视
     python build.py clean      # 清理构建
     python build.py fs         # 构建 + 上传文件系统

2. 版本号
   - 格式：v主.次.修订  例：v0.0.1
   - 从 v0.0.1 开始递增
   - 小功能/修复：修订号 +1
   - 新功能/较大改动：次版本 +1
   - 重大架构变更：主版本 +1

3. Git 提交
   - 提交说明用中文，清晰描述变更内容
   - 一个提交一个逻辑变更
   - 新增功能在分支开发，完成后合并到主分支
   - 不擅自删除历史、不强制覆盖、不执行破坏性回退


================================================================================
九、新增功能开发流程
================================================================================

1. 明确需求，确定属于哪一层
2. 在对应层创建 .h/.cpp 文件对
3. 如有需要，在 common/config.h 添加配置项
4. 如需 RTOS 任务，在 rtos/ 目录添加任务文件
5. 在 main.cpp（或 app_main）中初始化并创建任务
6. 编译验证：python build.py build
7. 上传测试：python build.py upload + monitor
8. 完成后提交 Git
